# Camera Border App Architecture Rules

本文件定义这个仓库的项目架构约束。目标不是追求“抽象完整”，而是避免各模块在早期就长歪。

适用范围：

- `electron/`
- `frontend/`
- `java-engine/`

## 1. 总体边界

这个项目是一个本地桌面应用，分为三个明确子系统：

- `electron/`：桌面壳。负责窗口生命周期、系统能力、启动和管理本地 Java 引擎。
- `frontend/`：工作台 UI。负责交互、状态、预览展示、任务状态展示。
- `java-engine/`：本地图像处理引擎。负责渲染、预览、导出、元数据处理。

硬性规则：

- `electron/` 不实现业务渲染逻辑。
- `frontend/` 不直接操作本地文件系统和 Java 进程。
- `java-engine/` 不关心 UI 布局和前端交互细节。
- 跨子系统通信必须通过明确的 API 或 preload 暴露接口完成，禁止隐式耦合。

## 2. 当前阶段目标

当前阶段是“先搭稳定框架，再填真实能力”。

允许先做：

- 工作台骨架
- Java 引擎生命周期
- 预览/渲染接口骨架
- 占位服务
- 基础请求校验

当前阶段不允许先做：

- ICC / 16-bit / TIFF
- 复杂模板系统
- 大而全的字体/数据库能力
- 为未来假想场景提前抽很多层

## 3. Repo 级结构规范

### 3.1 Electron

推荐结构：

```text
electron/src/
  main.ts
  preload.ts
  paths.ts
  java-bridge.ts
  runtime/
  ipc/
```

规则：

- `main.ts` 只负责应用启动、窗口创建、生命周期编排。
- `java-bridge.ts` 只负责 Java 引擎的启动、停止、健康探测、错误包装。
- `preload.ts` 只暴露经过筛选的安全运行时接口。
- 不要在 `main.ts` 内堆业务判断和大量字符串常量。

### 3.2 Frontend

推荐结构：

```text
frontend/src/
  App.tsx
  main.tsx
  styles.css
  api/
  components/
  features/
  types/
  state/
```

规则：

- `App.tsx` 只负责页面骨架组合，不承载大量业务逻辑。
- `api/` 只放请求封装和响应映射。
- `components/` 放可复用基础 UI。
- `features/` 放按业务分组的界面和状态逻辑，例如 `queue/`、`preview/`、`settings/`。
- `types/` 只放共享类型，不放行为逻辑。
- 前端禁止自行定义与后端长期契约冲突的“临时 DTO”。

### 3.3 Java Engine

推荐结构：

```text
java-engine/src/main/java/com/camborder/app/
  controller/
  model/
  service/
```

规则：

- `controller/` 只负责 HTTP 接口输入输出。
- `service/` 承载引擎行为，不把逻辑塞进 controller。
- `model/` 放请求、响应、领域配置对象。
- 占位实现必须放在 `Placeholder*Service`，避免和未来真实实现混在一起。

## 4. API 设计规则

### 4.1 单一渲染配方

预览和导出必须共享同一份核心渲染配置：

- 使用 `RenderRecipe` 作为共享核心模型。
- `PreviewRequest` 和 `RenderJobRequest` 只能在外围补充预览尺寸、导出设置等差异字段。
- 禁止为预览和导出维护两套意义相同但字段不一致的配置结构。

### 4.2 本地语义优先

这是桌面本地应用，字段命名必须反映本地资源语义：

- 用 `imagePath`
- 不用 `imageUrl`

除非后续真的支持远程资源，否则禁止引入 URL 语义。

### 4.3 面向 PRD 的最小正确模型

即使是 MVP，占位结构也必须朝真实目标对齐：

- 边框使用 `top/bottom/left/right`，不要只做单个 `width`
- 文本使用 `texts[]`，不要只做单个 `text`
- 导出设置单独建 `ExportSettings`
- 预览接口和导出接口都基于 `RenderRecipe`

### 4.4 接口风格

- `GET` 只用于纯读取且参数简单的接口
- 配置复杂的预览请求必须使用 `POST`
- 异步导出任务使用 job 风格接口，例如：
  - `POST /api/render-jobs`
  - `GET /api/render-jobs/{jobId}`

## 5. 校验与错误处理

### 5.1 请求校验

所有系统边界输入必须校验，至少包括：

- 必填字段
- 数值范围
- 枚举合法性
- 空 body

禁止把“空请求也返回 200”固化成测试用例。

### 5.2 错误语义

- 4xx：请求不合法
- 5xx：服务内部异常
- 占位实现可以返回受控占位结果，但不能吞掉明显的非法输入

## 6. 测试规则

每个新增接口至少覆盖两类测试：

- happy path
- invalid input / failure path

禁止只测“能通”。尤其是以下场景必须测：

- 空 body
- 缺少必填字段
- 非法枚举值
- 非法尺寸参数

## 7. 命名与抽象规则

### 7.1 什么时候可以抽象

只有在以下条件之一成立时才允许新增抽象层：

- 已出现第二个明确实现
- 已存在重复逻辑
- 不抽象会明显让边界混乱

否则先保持直接。

### 7.2 什么时候不要抽象

以下情况禁止过度设计：

- 为尚未实现的 V2/V3 能力提前铺很多接口
- 为单一占位实现引入过多工厂、策略、注册表
- 前端先做一整套复杂状态架构，但还没有真实数据流

## 8. 注释与说明语言

为了降低协作成本，仓库内新增的代码注释、说明性文案和面向开发者的错误提示统一使用简体中文。

适用范围：

- 代码注释
- 提交给协作 agent 的代码内说明
- 面向开发者的占位提示和 TODO 说明
- 新增架构文档、设计说明、开发说明

规则：

- 默认使用简体中文，不中英混杂表达同一层含义。
- 变量名、类型名、接口名、路径名仍按现有代码风格使用英文。
- 面向最终用户的界面文案可以根据产品需要使用英文或中英混合，但代码注释和开发说明仍保持简体中文。
- 引用外部协议字段或第三方库原始术语时，可保留英文原词，但解释说明使用简体中文。

## 9. Definition of Done

一个“框架搭建”类任务完成前，必须同时满足：

- 代码位置符合本文件结构约束
- 接口命名与字段命名符合本文件规则
- 至少有基础成功路径和失败路径测试
- 占位实现和未来真实实现边界清晰
- 没有把 UI 草稿直接硬编码成后端长期契约

## 10. Agent Working Rules

给执行 agent 的附加硬约束：

- 改代码前先判断改动属于 `electron`、`frontend` 还是 `java-engine`
- 不得跨子系统顺手重构
- 如果新增接口或 DTO，必须先检查是否违反本文件第 4 节
- 如果新增目录或分层，必须说明为什么需要这一层
- 如果测试只覆盖 happy path，任务不算完成
- 新增代码注释和说明默认使用简体中文

## 11. 当前推荐下一步

优先顺序如下：

1. 固定 Java API 骨架和请求模型
2. 固定 Electron 启动 Java 的生命周期
3. 固定前端工作台结构
4. 再接单图预览与导出闭环

不要跳过前 3 步直接做复杂业务能力。
